# IPC 鉴权安全检查清单（IPC_AUTH_CHECKLIST）

范围：OpenHarmony 系统框架中涉及 `IRemoteObject` / Stub / Proxy / `IPCSkeleton` / `OnRemoteRequest` / `SendRequest` / `WriteRemoteObject` / `GetSystemAbility` 的 C++、Rust、构建脚本及权限配置文件。

目标：**识别 IPC 调用链中的权限绕过、身份伪造、中继提权与校验逻辑缺陷**，作为 `security_review` 技能在 IPC 场景下的专项补充。

## 使用方式

1. 当被审计代码包含 IPC 接口实现、跨进程服务代理或系统能力代理时，本清单必须启用。
2. 按六个方向逐条核对；每条若存在反例，必须按 `security_review` 输出模板形成正式发现。
3. 对分布式/跨设备场景，额外关注 F.6.2。

---

## 一、校验位置类（位置错）

- [ ] **1.1 鉴权做在客户端，服务端裸奔**
  - 检查：Proxy 层校验后，Stub 层是否直接执行？攻击者可直接 `GetSystemAbility` 拼 `Parcel` 调 code 绕过。

- [ ] **1.2 多次 IPC 只覆盖第一跳**
  - 检查：入口 code 有鉴权，内部 code 是否无鉴权？攻击者可跳过入口直接发内部 code。

- [ ] **1.3 同一服务部分 code 漏挂鉴权**
  - 检查：调试/测试/遗留 code 是否未走鉴权分支？新接口是否只改了业务逻辑没登记鉴权矩阵？

## 二、身份来源类（身份错）

- [ ] **2.1 信任 Parcel 里自报的身份**
  - 检查：是否从 `data` 读取 `uid` / `bundleName` / `token` 而非使用 `IPCSkeleton::GetCallingUid()` / `GetCallingTokenID()`？

- [ ] **2.2 信任客户端传的"校验结果"**
  - 检查：是否直接读取 `isSystemApp`、`permissionGranted`、`callerType` 等布尔值做判断？

- [ ] **2.3 native/shell 身份直接放行的后门**
  - 检查：是否对 `TOKEN_NATIVE`、`ROOT_UID`、`SHELL_UID` 直接放行？是否 `isSubsystem` 即放行？

## 三、调用链 / 中转类（链路错）

- [ ] **3.1 SA 中继：服务端不认转发调用**
  - 检查：是否只校验直接调用方 token，未区分 `isProxy`、未对原始调用方二次授权？

- [ ] **3.2 服务端用自己的身份去调下游，替调用方提权**
  - 检查：是否调用 `ResetCallingIdentity()` / `clearCallingIdentity()` 后未恢复身份就鉴权？

## 四、分发与解析类（分发错）

- [ ] **4.1 未知 code 落进 default 放行 / 先执行后校验**
  - 检查：`switch` 的 `default` 分支是否有副作用？未知 code 是否静默执行公共逻辑？

- [ ] **4.2 校验失败但没有 return**
  - 检查：校验失败是否只打日志而未阻断流程？错误码比较是否写反（`==` vs `!=`）？是否用 `0` 判断成功而接口返回负值？

- [ ] **4.3 校验与执行解析不一致（反序列化绕过）**
  - 检查：校验时读取的字段与执行时反序列化的对象是否一致？`Parcel` 是否被重置位置后重新解析？

- [ ] **4.4 先产生副作用再鉴权**
  - 检查：是否存在 `WriteFile` / `StartAbility` / 发广播 先于鉴权执行的情况？

- [ ] **4.5 异常分支吞掉校验**
  - 检查：`try-catch` 中校验异常是否被捕获后直接 fallthrough 执行后续逻辑？

## 五、校验逻辑缺陷类（逻辑错）

- [ ] **5.1 黑名单式校验**
  - 检查：是否用黑名单过滤？大小写、编码、`/../`、符号链接、等价 Unicode 等变体是否可绕过？

- [ ] **5.2 前缀/包含匹配代替精确匹配**
  - 检查：是否用 `find()` / `starts_with()` 做包名/路径校验？如 `com.ohos.evil` 是否可通过？

- [ ] **5.3 只校验"能不能调"，不校验"对谁做"**
  - 检查：权限校验通过后，是否允许操作 Parcel 里任意指定的目标包名/路径？

- [ ] **5.4 参数组合校验不全**
  - 检查：是否只校验了 `action` 未校验 `uri`？校验了 `uri` 未校验 `extras` 里嵌套的二次意图/回调地址？

- [ ] **5.5 整数截断/溢出绕过范围检查**
  - 检查：`code`、`index`、`size` 是否用窄类型接收？`int32` 读 `uint32` 溢出后是否绕过 `index < MAX` 判断？

## 六、生命周期 / 配置类（状态错）

- [ ] **6.1 权限定义 `availableLevel` 配错**
  - 检查：自定义权限是否声明为 `normal` 级？普通 HAP 申请即得是否导致校验门槛形同虚设？

- [ ] **6.2 跨设备（分布式）需要校验真实调用方**
  - 检查：本端是否基于 `IPCSkeleton` 获取的真实调用方身份重新鉴权，而非直接采信对端设备传来的 `userId` / 包名 / 权限结论？对端设备本身是否完成可信认证，转发链路上的身份是否可被篡改？
